\section{Overview}

We are given longitudinal clinical notes per patient (2--13 notes), ordered chronologically by filename (\texttt{text\_0.md} earliest). The task is to extract a structured condition summary per patient and write one JSON file \texttt{patient\_XX.json} following the schema in \texttt{problem\_statement.md}. Each condition must:
\begin{itemize}[leftmargin=1.2em]
  \item map to exactly one taxonomy slot (\texttt{category.subcategory}) from \texttt{taxonomy.json};
  \item include a \texttt{status} in \{\texttt{active}, \texttt{resolved}, \texttt{suspected}\} reflecting the latest note where it appears;
  \item include an \texttt{onset} date using the specified priority rules (or \texttt{null});
  \item include comprehensive evidence excerpts across notes with exact \texttt{note\_id}, \texttt{line\_no}, and verbatim \texttt{span}.
\end{itemize}

\section{System Design}

The system is a \textbf{two-stage LLM extraction pipeline} with strict post-validation, deterministic evidence hardening, and fuzzy taxonomy recovery.

\subsection{Stage A: Note-Level Extraction}
For each clinical note:
\begin{enumerate}[leftmargin=1.2em]
  \item Load the note with 1-indexed line numbers.
  \item Construct a \textbf{rich system prompt} (\texttt{prompts.py}) containing:
    \begin{itemize}
      \item Full taxonomy with descriptions, subcategory explanations, and examples
      \item Status value definitions with signal keywords
      \item Disambiguation rules verbatim from taxonomy (heart failure categorization, diabetic complications)
      \item Onset priority rules and date format requirements
      \item A few-shot example demonstrating expected input/output format
    \end{itemize}
  \item Prompt the LLM to extract all taxonomy-valid conditions.
  \item \textbf{Coerce evidence spans}: enforce each \texttt{evidence.span} is an exact substring of the referenced \texttt{line\_no}; fall back to the full line if the LLM paraphrased.
  \item \textbf{Recover invalid taxonomy keys}: if the LLM outputs a close-but-wrong category/subcategory key, attempt fuzzy matching ($\geq$ 75\% similarity) against valid keys before discarding.
  \item Validate each condition against Pydantic schema + taxonomy keys.
\end{enumerate}

\subsection{Stage B: Patient-Level Consolidation}
All per-note candidates are consolidated into a single patient summary:
\begin{itemize}[leftmargin=1.2em]
  \item \textbf{Deduplication}: synonymous conditions within the same taxonomy slot are merged using fuzzy name matching ($\geq$ 94\% similarity).
  \item \textbf{Status}: uses the status from the chronologically latest note where the condition appears.
  \item \textbf{Onset}: uses the earliest non-null onset, following the taxonomy's priority rules.
  \item \textbf{Evidence}: the LLM is explicitly instructed to include evidence from every note where the condition is mentioned.
\end{itemize}

If the consolidation output fails validation, the system falls back to a \textbf{deterministic merge}: dedup by normalized name + taxonomy slot, earliest onset, latest status, union of evidence.

\subsection{Stage C: Evidence Hardening (Deterministic Post-Pass)}
After consolidation, a deterministic pass scans all note lines for missed evidence:
\begin{itemize}[leftmargin=1.2em]
  \item Matches the full condition name (case-insensitive) and known clinical abbreviations (e.g., ``ARDS'', ``COPD'', ``NIDDM'').
  \item Adds missing evidence entries conservatively (minimum 5-character term length to avoid noisy matches).
  \item Deduplicates evidence by \texttt{(note\_id, line\_no)} and verifies span exactness.
\end{itemize}

\subsection{Caching, Cost, and Determinism}
\begin{itemize}[leftmargin=1.2em]
  \item LLM calls cached on disk (SHA-256 hash of prompt content); re-runs avoid duplicate tokens.
  \item Default temperature 0.0 for determinism.
  \item Token usage tracked per call; summary printed at end of each run.
  \item Robust JSON parsing handles fenced code blocks and embedded JSON.
\end{itemize}

\section{Key Design Decisions}

\begin{enumerate}[leftmargin=1.2em]
  \item \textbf{Full taxonomy in prompts}: Including subcategory descriptions and examples ($\sim$15K char system prompt) substantially improves category assignment accuracy vs.\ keys-only.
  \item \textbf{Disambiguation rules verbatim}: Prevents common errors (e.g., diabetic nephropathy classified under renal instead of \texttt{metabolic\_endocrine.diabetes}).
  \item \textbf{Increased max\_output\_tokens (4096)}: Prevents truncation for patients with many conditions (e.g., patient\_16 has 23+ conditions).
  \item \textbf{Conservative evidence hardening}: Only matches full condition name and known abbreviations — does \emph{not} decompose names into individual words to avoid noisy false matches.
  \item \textbf{Schema relaxation}: \texttt{extra="ignore"} on Pydantic models so LLMs adding extra fields (e.g., ``confidence'') don't crash the pipeline.
\end{enumerate}

\section{What Worked / What Didn't}

\subsection{What Worked}
\begin{itemize}[leftmargin=1.2em]
  \item Two-stage extraction reduces missed conditions and encourages comprehensive evidence.
  \item Full taxonomy in prompts substantially improves category/subcategory accuracy.
  \item Evidence hardening deterministically fills evidence gaps for conditions mentioned across many notes.
  \item Fuzzy taxonomy recovery saves conditions that would otherwise be dropped due to minor LLM key typos.
  \item Disk caching and parallel processing improve speed and cost substantially on re-runs.
  \item Robust JSON parsing handles models that output fenced code blocks.
\end{itemize}

\subsection{Risks / Limitations}
\begin{itemize}[leftmargin=1.2em]
  \item Different models (GPT-4, Claude, Mixtral) have different instruction-following behaviors; prompts may need tuning for the evaluation model.
  \item Onset date extraction involving relative date conversion is heavily model-dependent.
  \item Rich taxonomy prompts ($\sim$15K char system prompt) increase per-call token usage, offset by improved accuracy.
\end{itemize}
